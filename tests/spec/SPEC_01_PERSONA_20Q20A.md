# FreeLang v4 — Spec 01: 타겟 페르소나 20Q 20A

---

## Q1. 이 언어를 쓰는 주체가 정확히 누구인가?

**A:** Claude Code.

더 넓게 말하면 LLM 기반 코드 생성 에이전트 전반이지만, 첫 번째 타겟은 Claude Code 하나다. 범용 타겟을 잡으면 v3처럼 흐려진다. 한 명의 사용자(Claude Code)가 만족하면, 다른 AI Agent도 자연스럽게 쓸 수 있다.

```
첫 번째 사용자: Claude Code (Anthropic CLI)
두 번째 사용자: 다른 AI Agent (향후)
세 번째 사용자: AI가 생성한 코드를 읽는 사람 (읽기 전용)
```

## Q2. Claude Code가 코드를 작성하는 실제 과정은?

**A:** 다음 루프를 반복한다.

```
1. 사용자가 자연어로 지시한다
   "파일 읽어서 에러 처리하는 함수 만들어줘"

2. Claude Code가 코드를 생성한다
   fn read_file(path: string): Result<string, Error> { ... }

3. 컴파일한다
   freelang build main.fl

4. 결과를 본다
   a) 컴파일 성공 → 실행
   b) 컴파일 에러 → 에러 메시지를 읽고 코드 수정 → 3번으로

5. 실행 결과를 확인한다
   a) 정상 → 완료
   b) 런타임 에러 → 코드 수정 → 3번으로
```

핵심: **4b 루프의 빈도를 최소화**하는 것이 이 언어의 목표다.
이상적으로는 4b가 0~2회 안에 끝나야 한다.

## Q3. 사람(사용자)의 역할은 무엇인가?

**A:** 의도를 말하고, 결과를 확인한다. 코드를 직접 작성하지 않는다.

```
사람: "사용자 인증 API 만들어줘"
      "에러 처리 빠진 것 같은데 확인해봐"
      "1000명 동시 접속 되게 해줘"

사람이 하지 않는 것:
      코드 직접 작성
      타입 명시
      메모리 관리
      디버깅 (AI가 함)
```

따라서 언어 문법이 "사람이 타이핑하기 편한가"는 중요하지 않다.
"AI가 생성하기 쉽고, 컴파일러가 검증하기 쉬운가"가 중요하다.

## Q4. Claude Code가 기존 언어를 쓸 때 실제로 겪는 문제는?

**A:** 4가지.

```
문제 1: 런타임에서야 발견되는 에러
  → Python에서 None.attribute 접근
  → JS에서 undefined is not a function
  → 컴파일(또는 실행 전 검증)에서 못 잡음

문제 2: 에러 경로 누락
  → try-catch를 빠뜨림
  → Go에서 err를 _ 로 무시
  → 행복한 경로만 작성하고 끝냄

문제 3: 동시성 코드의 비결정적 버그
  → 테스트에선 통과, 프로덕션에서 터짐
  → 데이터 경쟁은 "가끔" 발생해서 재현이 어려움
  → AI가 이걸 사전에 감지하는 건 불가능

문제 4: 같은 일을 하는 방법이 너무 많음
  → JS: callback / Promise / async-await / Observable
  → Python: threading / multiprocessing / asyncio / concurrent.futures
  → AI가 어떤 방식을 쓸지 매번 판단해야 함
  → 일관성 없는 코드 생성
```

## Q5. Claude Code가 잘하는 것은?

**A:**

```
잘하는 것:
  1. 패턴 인식: 비슷한 코드를 빠르게 생성
  2. 규칙 준수: 명확한 규칙이 있으면 정확히 따름
  3. 반복 작업: 보일러플레이트를 실수 없이 생성
  4. 에러 메시지 해석: 구체적인 에러면 즉시 수정 가능
  5. 대량 생성: 100개 함수를 일관된 스타일로 생성
```

언어 설계 시사점: **규칙이 명확하고, 에러 메시지가 구체적이면** AI는 거의 실수하지 않는다.

## Q6. Claude Code가 못하는 것은?

**A:**

```
못하는 것:
  1. 암묵적 규칙 추론: "보통은 이렇게 한다"를 놓침
  2. 비결정적 버그 감지: 레이스 컨디션을 사전에 발견 못함
  3. 맥락 없는 판단: "이 에러는 무시해도 된다"는 판단을 못함
  4. 복잡한 상태 추적: 10단계 넘는 상태 전이를 놓침
  5. Lifetime 같은 추상적 소유권 추론: Rust의 'a 를 정확히 쓰기 어려움
```

언어 설계 시사점: **암묵적인 것을 제거하고, 판단이 필요한 부분을 컴파일러가 강제**하면 AI의 약점을 보완할 수 있다.

## Q9. "안전하다"의 정확한 정의는?

**A:** 컴파일 통과 시 다음 5가지가 보장된다.

```
안전성 1: 타입 안전 (Type Safety)
  → 모든 값의 타입이 컴파일 시점에 확정
  → 묵시적 변환 없음
  → 잘못된 타입 사용 = 컴파일 에러

안전성 2: Null 안전 (Null Safety)
  → null/nil/undefined 존재하지 않음
  → 값이 없을 수 있으면 Option<T>
  → Option을 처리하지 않으면 컴파일 에러

안전성 3: 에러 완전성 (Error Completeness)
  → 실패할 수 있는 함수는 Result<T,E> 반환
  → Result를 처리하지 않으면 컴파일 에러
  → match의 모든 case 처리 강제

안전성 4: 메모리 안전 (Memory Safety)
  → use-after-free 불가능 (Ownership)
  → 이중 해제 불가능
  → 메모리 누수 컴파일 시점 감지

안전성 5: 동시성 안전 (Concurrency Safety)
  → 공유 가변 상태 없음
  → 데이터 경쟁 구조적 불가능
  → 채널로만 통신
```

## Q10. "편안하다"의 정확한 정의는?

**A:** AI가 코드를 생성할 때 **판단할 것이 적다**.

```
편안함 1: 선택지가 하나
  → 에러 처리 방법: Result 하나
  → 반복문: for 하나
  → 동시성: Actor + channel 하나
  → "어떤 방식을 쓸까?" 고민 없음

편안함 2: 에러 메시지가 수정 방법을 알려줌
  → "Type mismatch" ❌
  → "Expected Result<i32, Error>, got i32. Wrap with Ok(value)" ✅
  → AI가 에러 메시지를 복사해서 수정하면 됨

편안함 3: 보일러플레이트가 적음
  → 타입 추론 (변수 선언 시 타입 생략 가능)
  → ? 연산자 (에러 전파 한 줄)
  → 패턴 매칭 (분기 간결)

편안함 4: 예측 가능
  → 코드를 읽으면 동작이 명확
  → 숨겨진 부작용 없음
  → 실행 순서가 코드 순서와 일치
```

## Q11. Rust를 그냥 쓰면 안 되는가?

**A:** 안 된다. 이유 하나: **Lifetime.**

```rust
// Rust: AI가 이걸 정확히 써야 한다
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

Lifetime annotation은 "값이 얼마나 오래 살아있는가"를 사람이 명시하는 것이다. AI는 이걸 자주 틀린다. Rust 컴파일러의 에러 메시지도 Lifetime 관련은 난해하다.

FreeLang v4의 선택: **Lifetime을 없애고 컴파일러가 자동 추론한다.** 추론 실패 시 "이 값을 클론하세요"라고 안내한다. 성능을 약간 포기하더라도 안전성을 택한다.

## Q12. Go를 그냥 쓰면 안 되는가?

**A:** 안 된다. 이유 둘: **에러 무시 가능 + Null 존재.**

```go
// Go: 에러를 무시할 수 있다
result, _ := dangerousFunction()  // _ 로 에러 버림. 컴파일 통과.

// Go: nil이 존재한다
var p *User = nil
fmt.Println(p.Name)  // 런타임 패닉
```

Go의 에러 처리는 "관례"에 의존한다. 컴파일러가 강제하지 않는다.
AI는 관례를 놓친다. 컴파일러가 강제해야 한다.

---

## Q13. TypeScript를 그냥 쓰면 안 되는가?

**A:** 안 된다. 이유: **`any` 탈출구 + 런타임 보장 없음.**

```typescript
// TypeScript: any로 타입 시스템 무력화
const data: any = fetchSomething()
data.whatever.doesnt.exist  // 컴파일 통과. 런타임 폭발.

// TypeScript: 타입이 런타임에 존재하지 않음
// 컴파일 후 JavaScript가 되면 타입 정보가 사라짐
```

TypeScript의 타입 시스템은 "힌트"다. 강제가 아니다.
FreeLang v4의 타입 시스템은 강제다. 탈출구가 없다.

## Q15. 이 언어로 하지 않을 것은?

**A:** 명시적 비목표(Non-goals).

```
비목표 1: 모든 개발자가 쓸 수 있는 범용 언어
  → 타겟은 AI Agent. 사람의 타이핑 편의성은 우선순위 아님.

비목표 2: Rust 수준 성능
  → Go 수준이면 충분. 성능보다 안전성.

비목표 3: 기존 언어와의 호환성
  → C FFI 이외의 상호운용은 비목표. 독립적 생태계.

비목표 4: 점진적 타입 시스템 (Gradual Typing)
  → "일단 any로 쓰고 나중에 타입 붙이기" 없음.
  → 처음부터 전부 타입. 예외 없음.

비목표 5: 빠른 프로토타이핑
  → Python처럼 "대충 돌아가게" 하는 용도 아님.
  → 처음부터 올바르게.
```

## Q16. 컴파일러 구현 언어는 무엇으로 하는가?

**A:** 지금 결정한다.

| 후보 | 장점 | 단점 | 판정 |
|------|------|------|------|
| TypeScript | 내가(Claude) 가장 능숙, AST 다루기 좋음, 빠른 반복 | v2/v3에서 사용 (실패 경험) | ⚠️ |
| Rust | 컴파일러에 최적, 성능 최고 | 개발 속도 느림, Lifetime 이슈 | ❌ |
| Go | 단순, 빠른 컴파일, 동시성 내장 | AST 다루기 불편 (제네릭 부족했음) | ⚠️ |
| C | 전통적, 최고 성능 | 메모리 관리 부담, 개발 느림 | ❌ |
| Python | 빠른 프로토타이핑 | 너무 느림 (컴파일러로 부적합) | ❌ |

**결정: TypeScript**

이유:
1. v2/v3 실패 원인은 TypeScript가 아니라 **범위 초과**였다
2. AST 조작에 TypeScript의 유니온 타입이 자연스럽다
3. 내가(Claude) 가장 생산성이 높다
4. Node.js 생태계로 테스트/빌드 도구 즉시 사용 가능
5. v4가 완성된 후 self-hosting(FreeLang으로 FreeLang 컴파일러 작성)이 최종 목표

```
Phase 1-13: TypeScript로 구현 (Bootstrap)
Phase 14+:  FreeLang으로 self-hosting (향후)
```

## Q17. 배포 형태는?

**A:**

```
개발 시: freelang build main.fl → bytecode → VM 실행
배포 시: freelang build --release main.fl → 단일 바이너리 (향후 AOT)

v4 범위:
  ✅ Bytecode VM (인터프리터)
  ⚠️ 단일 바이너리 (v4 후반)
```

```bash
# 컴파일 + 실행
freelang run main.fl

# 컴파일만
freelang build main.fl → main.flc (bytecode)

# Bytecode 실행
freelang exec main.flc
```

## Q18. 에러 메시지 설계 원칙은?

**A:** AI가 읽고 자동으로 수정할 수 있어야 한다.

```
나쁜 에러 메시지:
  error[E0308]: mismatched types

좋은 에러 메시지:
  error[FL001]: 타입 불일치
    --> main.fl:12:5
    |
  12|   var x: i32 = "hello"
    |                 ^^^^^^^
    |
    = expected: i32
    = found:    string
    = fix: i32("hello") 또는 변수 타입을 string으로 변경
```

에러 메시지 3요소:
1. **위치**: 파일, 줄, 칸
2. **원인**: expected vs found
3. **수정 방법**: 구체적 코드 제안

AI는 `fix:` 줄을 읽고 자동 수정한다.

## Q19. 성공 기준 (구체적 수치)은?

**A:**

```
기준 1: 악마 테스트
  → 악마 테스트 #2 (채널 + Result + spawn): 통과
  → 악마 테스트 #3 (에러 전파 + 반복): 통과
  → 1000개 spawn, 데이터 경쟁 0: 통과

기준 2: 컴파일 → 런타임 안전성
  → 컴파일 통과한 코드의 런타임 에러 발생률: 0%
  → (panic 제외. panic은 의도적 중단)

기준 3: AI 자동 수정 루프
  → 컴파일 에러 → AI 자동 수정 → 재컴파일 성공: 3회 이내

기준 4: 규모
  → 컴파일러 코드: 45,000 LOC 이내
  → 테스트: 3,000+ (99% 통과)
  → 컴파일 시간: 1,000줄 파일 1초 이내
```

## Q20. v1, v2, v3의 실패 원인을 이번에 어떻게 피하는가?

**A:**

```
v3 실패 원인: 설계 90%, 구현 40%. 야심과 현실의 갭.
  → 24개 Phase를 계획하고, 12개까지만 겨우 구현.
  → 동시성이 종이 위에만 존재.

v2 실패 원인: Phase 1-11까지 기초는 됐지만 범위 초과.
  → 12,600 LOC까지 갔지만 핵심(동시성, 안전성) 미도달.

이번에 다르게 하는 것:

1. 설계를 먼저 완성한다 (1부 10단계 명세서)
   → 코드 치기 전에 "뭘 만드는지" 확정
   → v3은 만들면서 설계했다. v4는 설계 후 만든다.
```
